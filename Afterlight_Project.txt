1. ОБЩЕЕ ОПИСАНИЕ ПРОЕКТА
===============================================================================

Afterlight - это локальное desktop-приложение для Windows, предназначенное для
ведения задач, заметок, категорий и повседневного рабочего расписания. Проект
ориентирован на быстрый личный workflow: пользователь открывает приложение,
создает задачи, распределяет их по разделам, следит за сроками, работает с
заметками и может добавлять задачи через Telegram-бота, не открывая основное
окно приложения.

Главная идея проекта - совместить простоту классического task manager с
удобством локального приложения:

- данные хранятся локально в SQLite;
- приложение работает без обязательного сервера;
- интерфейс адаптирован под светлую и темную темы;
- основные сценарии доступны через боковую панель, вкладки и поиск;
- Telegram-бот позволяет добавлять, просматривать, завершать и удалять задачи
  из мессенджера;
- системные Windows-уведомления и Telegram-уведомления помогают не пропускать
  дедлайны.

MVP проекта уже покрывает ключевые пользовательские сценарии:

- первичная настройка профиля;
- создание, редактирование, удаление и выполнение задач;
- работа с разделами "Входящие", "Сегодня", "Неделя" и пользовательскими
  категориями;
- ведение заметок на страницах;
- поиск и история переходов;
- настройки интерфейса, темы, уведомлений, боковой панели, Telegram и резервных
  копий;
- импорт/экспорт задач;
- локальная интеграция с Telegram-ботом.

2. ИСТОРИЯ РАЗВИТИЯ И ПОДХОД К РАЗРАБОТКЕ
===============================================================================

Проект развивался поэтапно.

2.1. Дизайн-макет

На первом этапе была сформирована визуальная концепция приложения в Figma:
расположение titlebar, боковой панели, рабочей области, вкладок, popup-окон,
карточек задач, настроек и системных элементов. Макет определил визуальную
основу Afterlight: строгий desktop-интерфейс, крупная навигация, отдельные
разделы задач, компактная панель действий и поддержка светлой/темной темы.

2.2. Первичный экспорт HTML/CSS

После макета часть интерфейсной структуры была перенесена в HTML/CSS-вид. Этот
этап был полезен для быстрой проверки размеров, отступов, иконок и визуального
баланса. Затем статическая верстка постепенно превращалась в полноценное
React-приложение: появились компоненты, состояние, обработчики событий,
диалоги, формы, работа с данными и IPC-слой.

2.3. Переход к Electron-приложению

Для desktop-версии выбран Electron. Он позволил совместить web-интерфейс на
React с возможностями нативного приложения:

- собственное окно без стандартной рамки;
- кастомный titlebar;
- системный трей;
- Windows-уведомления;
- локальные файлы и SQLite;
- preload API для безопасного взаимодействия renderer и main процессов;
- упаковка приложения через Electron Forge.

2.4. Локальное хранение данных

После появления базовой UI-логики проект был переведен на локальное хранение в
SQLite. Это сделало данные постоянными между запусками, позволило ввести
профиль, workspace, категории, задачи, заметки и настройки.

2.5. Развитие UX

В дальнейшем были доработаны:

- темная тема;
- анимации popup, dropdown, вкладок, меню, sidebar и переходов;
- поиск с историей и недавно открытыми разделами;
- markdown-контент для changelog и help;
- внешний файл версии приложения;
- более аккуратные hover-эффекты;
- адаптация иконок под тему;
- Telegram-интеграция.

2.6. Telegram-бот

Telegram-бот был добавлен как демонстрационная и практически полезная
интеграция. В текущем варианте он работает без сервера: пока запущен
Afterlight, main-процесс приложения запускает polling к Telegram Bot API.
Пользователь может управлять задачами из Telegram, а приложение может отправлять
напоминания о дедлайнах.

4. ОБЩАЯ АРХИТЕКТУРА
===============================================================================

Проект построен по классической Electron-схеме:

1. Main process
   Отвечает за окно приложения, системный трей, файловую систему, SQLite,
   экспорт/импорт, уведомления Windows и Telegram-бота.

2. Preload process
   Создает безопасный мост между renderer и main через contextBridge. Renderer
   не получает прямой доступ к Node.js, а вызывает только явно разрешенные IPC
   методы.

3. Renderer process
   React-интерфейс приложения. В нем находятся страницы, popup-окна, настройки,
   store, локализация и визуальная логика.

4. Shared layer
   Общие TypeScript-типы и файл версии, доступные нескольким частям проекта.

Структура src:

- src/main/main.ts
  Точка входа main-процесса. Создает BrowserWindow, регистрирует системные IPC,
  управляет tray, уведомлениями, резервными копиями, импортом/экспортом и
  запуском Telegram-бота.

- src/main/preload.ts
  Безопасный API для renderer. Через window.afterlightApi прокидываются методы
  работы с задачами, категориями, профилем, настройками, файлами, Telegram,
  системными действиями окна и событиями.

- src/main/ipc/tasks.ts
  Регистрация IPC-обработчиков для прикладных данных: загрузка app data,
  создание/редактирование/удаление задач и категорий, заметки, профиль,
  workspace, настройки.

- src/main/storage/database.ts
  Инициализация SQLite, создание таблиц, миграции, seed профиля и workspace.

- src/main/storage/repositories.ts
  Репозиторный слой. Здесь находится вся бизнес-логика CRUD, нормализация
  входных данных, default settings, обновление просроченности задач и доступ к
  данным приложения.

- src/main/telegram/bot.ts
  Локальный Telegram-бот: конфиг, polling, обработка сообщений, inline/reply
  keyboard, парсинг текста задач, категории, локализация, очистка чата,
  уведомления о дедлайнах.

- src/shared/types.ts
  Общие типы: Task, Category, Note, UserProfile, Workspace, AppSettings,
  TelegramBotStatus и другие.

- src/shared/app-version.json
  Единый файл версии приложения. Сейчас содержит "0.2.0". Интерфейс настроек
  читает версию из этого файла.

- src/renderer/main.tsx
  Точка входа React-приложения.

- src/renderer/App.tsx
  Главный компонент renderer. Собирает titlebar, sidebar, content, dialog
  layers, быстрые системные действия и bootstrap приложения.

- src/renderer/store/useTaskStore.ts
  Zustand-store. Хранит текущие задачи, категории, заметки, профиль, workspace,
  настройки, открытые вкладки, историю маршрутов, active scope/category и
  методы обновления данных через IPC.

- src/renderer/components/*
  Компоненты интерфейса.

- src/renderer/styles/app.css
  Основной CSS-файл: layout, темы, анимации, popup, настройки, sidebar,
  workspace, tab-bar, search, task dialog, category dialog, loading и другие
  визуальные состояния.

- src/renderer/content/*.md
  Внешний markdown-контент для help и changelog на русском и английском языках.

- assets/*
  SVG/PNG/ICO ресурсы: логотипы, иконки разделов, кнопки, статусы Telegram,
  чекбоксы, аватары, popup-иконки и другие визуальные ассеты.

5. ДАННЫЕ И МОДЕЛЬ ПРЕДМЕТНОЙ ОБЛАСТИ
===============================================================================

5.1. Task

Задача содержит:

- id - уникальный идентификатор;
- title - название;
- description - опциональное описание;
- dueDate - дата дедлайна в формате YYYY-MM-DD;
- dueLabel - время дедлайна в формате HH:mm;
- priority - приоритет 1..4;
- status - active или completed;
- scope - inbox, today, week или category;
- categoryId - ссылка на категорию;
- isExpired - признак просроченности;
- updatedAt - дата последнего изменения.

Особенности:

- задачи без даты могут жить во "Входящих" или в категории;
- задачи с сегодняшней датой могут отображаться на странице "Сегодня";
- задачи текущей недели отображаются на странице "Неделя";
- просроченные активные задачи могут подниматься вверху страницы "Сегодня";
- выполнение задачи меняет status на completed, но не удаляет запись.

5.2. Category

Категория содержит:

- id;
- title;
- color;
- emoji;
- iconMode: color, emoji или hash;
- isFavorite;
- updatedAt.

Категории могут отображаться в sidebar в двух группах: избранные и обычные.
Иконка категории может быть цветной точкой, символом # или emoji.

5.3. Note

Заметка привязана к области:

- scope;
- categoryId для заметки категории;
- text;
- updatedAt.

Заметки редактируются прямо на страницах, имеют автосохранение и лимит строк,
который настраивается пользователем.

5.4. UserProfile

Профиль содержит:

- id;
- name;
- email;
- avatarDataUrl;
- activeWorkspaceId;
- isSetupComplete.

Пароль хранится в БД в виде SHA-256 hash. Для MVP это демонстрационный механизм
локальной безопасности, а не полноценная промышленная система авторизации.

5.5. Workspace

Workspace содержит:

- id;
- title;
- profileId.

Сейчас приложение работает с одним активным workspace. Архитектура уже содержит
workspace_id в основных таблицах, поэтому проект можно развивать в сторону
нескольких рабочих пространств.

5.6. AppSettings

Настройки включают:

- тему интерфейса;
- язык;
- стартовый раздел;
- восстановление вкладок;
- состояние окна;
- запуск вместе с Windows;
- запуск свернутым;
- поведение при закрытии;
- tray;
- настройки страницы "Сегодня";
- сортировки задач и категорий;
- подтверждения удаления;
- счетчики;
- отображение tab-bar;
- автосохранение заметок;
- уведомления;
- резервные копии.

Новые настройки уведомлений версии 0.2.0:

- deadlineNotifyBeforeMinutes - за сколько минут до дедлайна уведомлять в
  Windows и Telegram. Значение по умолчанию: 60.
- todayRefreshNotifyBeforeMinutes - за сколько минут до обновления страницы
  "Сегодня" уведомлять в Windows. Значение по умолчанию: 10.
- overdueNotifyEveryMinutes - интервал повторных Windows-уведомлений о
  просроченных задачах. Значение по умолчанию: 240.

6. ОРГАНИЗАЦИЯ БАЗЫ ДАННЫХ
===============================================================================

SQLite база создается в пользовательской директории Electron:

app.getPath('userData')/storage/afterlight.sqlite

Рядом находятся:

- storage/backups - резервные копии SQLite;
- storage/telegram.json - локальный конфиг Telegram-бота.

6.1. Таблица profiles

Назначение: хранение локального профиля пользователя.

Поля:

- id TEXT PRIMARY KEY
- name TEXT NOT NULL
- email TEXT
- avatar_data_url TEXT
- password_hash TEXT
- is_setup_complete INTEGER NOT NULL DEFAULT 0
- active_workspace_id TEXT NOT NULL
- created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
- updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP

6.2. Таблица workspaces

Назначение: рабочие пространства пользователя.

Поля:

- id TEXT PRIMARY KEY
- profile_id TEXT NOT NULL
- title TEXT NOT NULL
- created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
- updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP

6.3. Таблица categories

Назначение: пользовательские категории задач.

Поля:

- id TEXT PRIMARY KEY
- workspace_id TEXT NOT NULL
- title TEXT NOT NULL
- color TEXT NOT NULL
- icon_mode TEXT NOT NULL DEFAULT 'color'
- emoji TEXT
- is_favorite INTEGER NOT NULL DEFAULT 0
- created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
- updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP

6.4. Таблица tasks

Назначение: задачи.

Поля:

- id TEXT PRIMARY KEY
- workspace_id TEXT NOT NULL
- title TEXT NOT NULL
- description TEXT
- due_date TEXT
- due_at TEXT
- priority INTEGER NOT NULL DEFAULT 4
- status TEXT NOT NULL DEFAULT 'active'
- scope TEXT NOT NULL DEFAULT 'inbox'
- category_id TEXT
- is_expired INTEGER NOT NULL DEFAULT 0
- created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
- updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP

6.5. Таблица notes

Назначение: заметки для разделов и категорий.

Поля:

- id TEXT PRIMARY KEY
- workspace_id TEXT NOT NULL
- scope TEXT NOT NULL
- category_id TEXT
- text TEXT NOT NULL DEFAULT ''
- updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP

6.6. Таблица app_settings

Назначение: настройки приложения на workspace.

Поля:

- workspace_id TEXT PRIMARY KEY
- settings_json TEXT NOT NULL
- updated_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP

Настройки хранятся JSON-объектом. При чтении применяется normalizeSettings:
старые настройки дополняются новыми default-полями, числовые значения
ограничиваются допустимыми диапазонами, enum-поля нормализуются.

6.7. Индексы

Создаются индексы:

- idx_tasks_scope;
- idx_tasks_category;
- idx_tasks_due_date;
- idx_notes_scope_category;
- idx_tasks_workspace;
- idx_categories_workspace;
- idx_notes_workspace.

Индексы ускоряют выборки задач по разделу, категории, дате и workspace.

6.8. Миграции

database.ts содержит простые миграции через PRAGMA table_info и ALTER TABLE.
Это позволяет обновлять существующую локальную базу пользователя без удаления
данных. Уже поддержаны миграции для password_hash, is_setup_complete,
is_expired, due_date, workspace_id, icon_mode, emoji, updated_at и category_id.

===============================================================================
7. MAIN PROCESS
===============================================================================

main.ts отвечает за системную часть приложения.

7.1. Окно приложения

Создается BrowserWindow:

- размер 1280x760;
- minWidth 1024;
- minHeight 640;
- frame: false;
- titleBarStyle: hidden;
- contextIsolation: true;
- nodeIntegration: false;
- sandbox: false;
- devTools: true.

Окно кастомное: стандартная рамка Windows заменена собственным titlebar.
DevTools открываются автоматически в development-режиме через
openDevToolsInDevelopment.

7.2. Single instance lock

app.requestSingleInstanceLock предотвращает запуск нескольких экземпляров.
Если пользователь запускает приложение повторно, текущий экземпляр получает
фокус. Это важно и для Telegram-бота: два polling-экземпляра одного бота
вызывают Telegram Conflict.

7.3. Tray

Tray включается настройками. Контекстное меню содержит:

- открыть Afterlight;
- добавить задачу;
- открыть "Сегодня";
- открыть "Неделю";
- выйти.

Действия из tray отправляются в renderer через system:quick-action.

7.4. Закрытие приложения

Поведение зависит от настроек:

- exit - закрыть приложение;
- ask - спросить подтверждение;
- tray - скрыть окно в системный трей.

7.5. Импорт и экспорт

Поддерживаются:

- export JSON;
- export CSV;
- import JSON;
- открытие папки данных;
- открытие расположения базы данных;
- ручное создание backup.

7.6. Windows-уведомления

Main process проверяет уведомления раз в минуту. В версии 0.2.0 логика
использует пользовательские интервалы:

- дедлайны - за deadlineNotifyBeforeMinutes минут;
- обновление "Сегодня" - за todayRefreshNotifyBeforeMinutes минут;
- просроченные задачи - повтор каждые overdueNotifyEveryMinutes минут.

Для предотвращения повторов используется набор notifiedKeys. Для дедлайнов ключ
включает id задачи, дату, время и интервал. Для просрочки используется временной
bucket на основе интервала. Для обновления "Сегодня" ключ включает дату
следующего refresh, время refresh и интервал уведомления.

7.7. Telegram-уведомления

При включенном notifyDeadlines main process дополнительно отправляет
Telegram-уведомление о дедлайне через notifyTelegramDeadline. Telegram-ключ
отдельный от Windows-ключа. Если отправка не удалась из-за сети, ключ удаляется,
и приложение попробует отправить напоминание на следующем проходе.

===============================================================================
8. PRELOAD И IPC
===============================================================================

Renderer не обращается к Node.js напрямую. Все системные возможности доступны
через window.afterlightApi, объявленный в preload.ts.

Основные группы методов:

- управление окном: minimize, toggleMaximize, setFullScreen, close;
- данные: getAppData, createTask, updateTask, deleteTask, deleteTasks,
  toggleTask;
- категории: createCategory, updateCategory, deleteCategory,
  toggleCategoryFavorite;
- заметки: updateNote;
- профиль и workspace: updateProfile, updateWorkspace, completeProfileSetup,
  resetProfile;
- настройки: updateSettings;
- файлы: exportTasksJson, exportTasksCsv, importTasksJson, createBackup,
  openDataFolder, openDatabase;
- Telegram: getTelegramStatus, configureTelegram, testTelegram,
  disconnectTelegram;
- события: onQuickAction, onDataChanged.

IPC-обработчики находятся в src/main/ipc/tasks.ts и системно в main.ts.
После изменения данных main может отправить событие system:data-changed, чтобы
renderer перезагрузил данные из БД.

9. RENDERER И UI
===============================================================================

9.1. App.tsx

Главный React-компонент собирает приложение:

- проверяет завершена ли первичная настройка профиля;

===============================================================================
12. TELEGRAM-ИНТЕГРАЦИЯ
===============================================================================

12.1. Общий принцип

Telegram-бот работает локально. Сервер не поднимается. Пока Afterlight запущен,
main process запускает polling к Telegram Bot API методом getUpdates.

Преимущество для MVP и дипломной демонстрации:

- не нужен VPS;
- не нужен публичный webhook URL;
- не нужен ngrok;
- все данные остаются локально;
- проще демонстрировать связку desktop app + Telegram.

Ограничение:

- бот работает только пока запущен Afterlight;
- нельзя запускать несколько экземпляров приложения с одним bot token;
- если приложение выключено, Telegram-сообщения не будут обработаны до
  следующего запуска.

12.2. Конфигурация

Конфиг хранится в:

app.getPath('userData')/storage/telegram.json

Содержит:

- token;
- enabled;
- chatId;
- botUsername;
- language;
- lastUpdateId;
- conversation;
- botMessageIds.

Токен не хранится в репозитории. Пользователь вводит его в настройках.

12.3. Подключение

Сценарий:

1. Пользователь создает бота через BotFather.
2. Вставляет token в настройках Afterlight.
3. Включает локального Telegram-бота.
4. Пишет /start боту.
5. Afterlight сохраняет chatId.
6. Бот отвечает главным меню.

Страница "Интеграция Telegram" показывает:

- статус polling;
- наличие токена;
- username бота;
- chat_id;
- большую SVG-иконку статуса;
- локальный token input;
- включение/отключение;
- автопроверку подключения каждые 5 секунд.

Кнопка "Проверить подключение" удалена: проверка выполняется автоматически.

12.4. Меню бота

Бот использует reply keyboard и inline keyboard.

Главное меню:

- Добавить задачу;
- Входящие;
- Сегодня;
- Неделя;
- Категории;
- Категория;
- Язык;
- Помощь.

В местах, где пользователь может выйти из сценария, есть кнопка:

❌️ Отмена

12.5. Команды

Поддерживаются:

- /start;
- /menu;
- /add;
- /inbox;
- /today;
- /week;
- /categories;
- /language;
- /help.

Команды регистрируются через setMyCommands и локализуются по выбранному языку.

12.6. Локализация

Бот поддерживает русский и английский языки. Язык хранится в telegram.json.
Пользователь может переключить язык кнопкой "Язык / Language".

12.7. Создание задач через Telegram

Пользователь может:

- отправить обычный текст;
- нажать "Добавить задачу";
- использовать /add текст.

Парсер понимает:

- даты: сегодня, завтра, today, tomorrow, 12.06, 12.06.2026;
- время: 09:30, 18:00;
- категорию через #Название;
- обычный остаток текста как title.

Примеры:

- Купить молоко
- Купить молоко сегодня
- Купить молоко завтра 18:00
- Сдать отчет 12.06 14:30
- Позвонить клиенту #Работа
- Домашка завтра #Учеба

Если категория указана через хэштег, но не найдена, бот предлагает создать
категорию или выбрать существующую.

12.8. Создание категорий через Telegram

Бот поддерживает упрощенное создание категории:

1. Пользователь нажимает "Категория".
2. Бот просит отправить название.
3. Название нормализуется.
4. Если категории нет, создается новая категория.
5. Цвет выбирается из предустановленного набора CATEGORY_COLORS.
6. iconMode по умолчанию hash.

12.9. Управление задачами

В списках задач бот показывает до 8 активных задач. Для каждой задачи есть:

- кнопка выполнить/вернуть;
- кнопка удалить.

После изменения данных вызывается onDataChanged, и renderer может обновить
локальное состояние.

12.10. Очистка чата

Бот запоминает id своих сообщений в botMessageIds и при действиях пользователя
пытается удалить старые сообщения. Это делает чат чище и оставляет пользователю
актуальное меню или актуальный результат действия.

Удаление сообщений выполняется best effort: Telegram может запретить удаление
очень старых или уже удаленных сообщений, поэтому ошибки deleteMessage
игнорируются.

12.11. Уведомления о дедлайнах в Telegram

Main process отслеживает дедлайны. Если notifyDeadlines включен, задача активна,
имеет dueDate и dueLabel, а до дедлайна осталось не больше
deadlineNotifyBeforeMinutes минут, приложение отправляет сообщение в Telegram.

Сообщение содержит:

- заголовок "Скоро дедлайн" / "Deadline soon";
- название задачи;
- дату и время;
- за сколько минут выполнено напоминание;
- inline-кнопки выполнить/удалить/перейти по меню.

Если Telegram-сеть недоступна, отправка помечается как неуспешная, и приложение
попробует снова на следующем проходе уведомлений.

16. БЕЗОПАСНОСТЬ И ОГРАНИЧЕНИЯ
===============================================================================

16.1. Локальные данные

Данные хранятся локально на компьютере пользователя. Приложение не отправляет
БД на внешний сервер.

16.2. Telegram token

Telegram token хранится в userData/storage/telegram.json. Его нельзя коммитить
в репозиторий. В будущем для production-версии стоит добавить:

- шифрование токена через safeStorage Electron;
- возможность удалить token полностью;
- проверку прав доступа к файлу;
- предупреждения при экспорте конфигов.

16.3. Пароль профиля

Пароль сейчас хэшируется SHA-256. Для дипломного MVP этого достаточно как
демонстрации локального поля безопасности. Для production нужна более надежная
схема: salt, bcrypt/argon2, безопасная смена пароля и, возможно, шифрование БД.

16.4. Telegram polling

Long polling одного бота не допускает два активных экземпляра. Проект снижает
риск через single instance lock, но конфликт все равно возможен, если тот же
token используется где-то еще.

19. ПОТЕНЦИАЛЬНЫЙ ВЕКТОР РАЗВИТИЯ

19.1. Серверная часть

Следующий крупный этап - сервер. Он может решать задачи:

- облачное хранение данных;
- синхронизация между несколькими устройствами;
- backup в облако;
- авторизация;
- Telegram webhook вместо local polling;
- push-уведомления;
- web-версия приложения;
- командная работа.

Возможный стек:

- Node.js/NestJS или Fastify;
- PostgreSQL;
- Prisma или Drizzle;
- Redis для очередей и фоновых задач;
- JWT/session auth;
- WebSocket для live sync;
- object storage для аватаров и вложений.

19.2. Telegram webhook

Если появится сервер, Telegram-бот лучше перевести с getUpdates на webhook.
Тогда бот сможет работать даже когда desktop-приложение закрыто. Сервер будет
принимать сообщения Telegram, создавать задачи в облачной БД, а desktop-клиент
будет синхронизироваться.

19.3. Облачная синхронизация

Можно ввести модель:

- локальная SQLite остается offline-cache;
- серверная БД является источником истины;
- изменения синхронизируются через очередь операций;
- конфликты решаются по updatedAt, version или CRDT-подходу.

19.4. Несколько workspace

20. ЗАКЛЮЧЕНИЕ
===============================================================================

Afterlight версии 0.2.0 представляет собой цельный MVP локального Windows task
manager. Приложение уже имеет полноценный desktop UI, постоянное локальное
хранение, настройки, темы, поиск, заметки, backup, markdown-документацию help и
changelog, а также глубокую Telegram-интеграцию без необходимости поднимать
сервер.

Главная архитектурная особенность проекта - разделение ответственности:

- Electron main отвечает за систему, БД, файлы, уведомления и Telegram;
- preload изолирует renderer от прямого доступа к Node.js;
- React renderer отвечает за интерфейс;
- Zustand-store связывает UI с данными;
- repositories централизуют работу с SQLite;
- shared types удерживают единый контракт данных.

Такой фундамент позволяет развивать проект дальше: от локального MVP к
